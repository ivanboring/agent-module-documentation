# Permissions

Dynamically generated by `NodePermissions::nodeTypePermissions()` (permission_callbacks). Which
appear depends on the settings toggles ([../configure/settings.md](../configure/settings.md)).
For type-specific ones, `<type>` is the node-type machine name (e.g. `article`).

## General (when `general_permissions` on)
| Permission | Reveals on the node form |
|---|---|
| `override all published option` | Published checkbox (`status` field) |
| `override all promote to front page option` | Promoted checkbox (`promote`) |
| `override all sticky option` | Sticky checkbox (`sticky`) |
| `override all revision option` | "Create new revision" checkbox |
| `enter all revision log entry` | Revision log message field |
| `override all authored by option` | Authored by / author (`uid`) |
| `override all authored on option` | Authored on date (`created`) |

## Content-type specific (when `specific_permissions` on)
Same seven, per type: `override <type> published option`,
`override <type> promote to front page option`, `override <type> sticky option`,
`override <type> revision option`, `enter <type> revision log entry`,
`override <type> authored on option`, `override <type> authored by option`.

## How it works
- `hook_entity_field_access_alter()` grants access to the `status`, `promote`, `sticky`, `uid`,
  `created`, and revision-log fields when the matching permission is held (Grant*Access classes
  in `src/Access/FieldAccess/`).
- `hook_form_alter()` (`GrantRevisionAccess`) exposes the revision checkbox.
- These only reveal options on nodes the user can already **edit**; they don't grant edit access.
  A user with `administer nodes` sees everything regardless.

Grant via drush:
```
drush role:perm:add editor 'override article published option'
```
