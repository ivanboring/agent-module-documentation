Mailer Plus (symfony_mailer) replaces Drupal's legacy mail system with a modern, object-oriented pipeline built on the Symfony Mailer component, producing themeable HTML emails with configurable policies, pluggable transports, and CSS inlining.

---

Drupal core's `MailManager` and `hook_mail` API are string-oriented, plain-text-first, and awkward to theme. Mailer Plus supersedes them with a structured `Email` object that flows through defined phases (init → build → pre-render → post-render → post-send), each of which modules and processors can act on. Emails are produced by **component mailers** — plugins declared with the `#[MailerInfo]` attribute and discovered by the `MailerLookup` manager — one per source (user, contact, commerce order, simplenews, update status, verify, etc.), so every email type is identified by a hierarchical tag like `user__password_reset`. Cross-cutting behaviour is applied by **email processors** (service-collected via the `symfony_mailer.add_processor` tag) and, through the Mailer Policy submodule, by configurable **EmailAdjuster** plugins (From, To, Subject, theme, inline CSS, convert to plain text, skip sending, choose transport, and more). Real emails are HTML, rendered through a Drupal theme and inlined with `css-to-inline-styles`, with a plain-text alternative generated by `html2text`. Delivery goes through pluggable transports (SMTP, Sendmail, native, DSN, null) configured by the Mailer Transport submodule, and legacy `hook_mail` emails can be transparently redirected into the new system by the Mailer Override submodule. Developers get fine-grained `hook_mailer_*` phase hooks, an alter hook for mailer definitions, and a clean `Email` API for building and sending mail in code. A verify screen at Configuration → System → Mailer Plus checks that outgoing email works.

---

- Send fully HTML, themeable emails instead of Drupal's plain-text defaults.
- Automatically inline CSS so styled emails render in email clients.
- Generate a plain-text alternative part from the HTML body automatically.
- Configure the From/Reply-To/Sender address for all site email centrally.
- Route outgoing mail through SMTP, Sendmail, a DSN string, or a null transport.
- Verify that email delivery works from the Mailer Plus settings page.
- Theme a specific email type (e.g. password reset) with its own template.
- Add a BCC address to every outgoing email for archival/compliance.
- Skip sending certain emails conditionally (e.g. to blocked users) via a processor.
- Override core user account emails (registration, password reset, status changes).
- Override contact form emails with rich HTML and per-form policies.
- Send Commerce order confirmation emails as branded HTML (with mailer_override).
- Integrate Simplenews newsletter and subscription emails into the pipeline.
- Send update-status security notification emails through the new mailer.
- Build and send a custom email programmatically via the Email API.
- Act on an email in a specific phase with `hook_mailer_TYPE_PHASE()`.
- Reuse an EmailAdjuster in code to tweak an email on the fly.
- Alter available mailer definitions with `hook_symfony_mailer_info_alter()`.
- Add tokens/variables to email subject and body fields per email type.
- Set email priority headers via an adjuster.
- Convert an HTML email to plain text for recipients who prefer it.
- Attach files to a generated email through the Email object.
- Centralise all email configuration as exportable Drupal config entities.
- Migrate a site off contrib SMTP/Mimemail modules onto a maintained stack.
- Gate email administration behind the `administer mailer` permission.
- Group emails by metadata (e.g. per contact form) for separate configuration.
