# Fix 404 pages — report, settings, API

## Report UI

| Route | Path | Purpose |
|---|---|---|
| `redirect_404.fix_404` | `/admin/config/search/redirect/404` | List logged 404s by hit count; add a redirect or ignore each (`administer redirects`) |
| `redirect_404.ignore_404` | `.../404/ignore` | Ignore a path (`administer redirect settings` **or** `ignore 404 requests`) |

"Add redirect" links to Redirect's add form with the source path pre-filled. "Ignore" marks a
row resolved so it drops off the list.

## Settings — `redirect_404.settings`

Schema `config/schema/redirect_404.schema.yml`. Defaults:

| Key | Default | Meaning |
|---|---|---|
| `row_limit` | `10000` | Max 404 rows to keep; older ones purged on cron |
| `pages` | `''` | Newline list of path patterns to **exclude** from logging |
| `suppress_404` | `false` | If true, don't write routine 404s to the standard logger |

## Storage service — `redirect.not_found_storage`

`Drupal\redirect_404\RedirectNotFoundStorageInterface` (impl `SqlRedirectNotFoundStorage`,
`backend_overridable`):

- `logRequest($path, $langcode)` — record/increment a 404.
- `resolveLogRequest($path, $langcode = NULL)` — mark handled (redirect added).
- `listRequests(array $header = [], $search = NULL)` — rows for the report.
- `purgeOldRequests()` — enforce `row_limit` (called on cron).
- `resetDailyCount()` — reset the daily hit counters.

`Redirect404LogSuppressor` decorates `logger.factory` to honor `suppress_404`.
